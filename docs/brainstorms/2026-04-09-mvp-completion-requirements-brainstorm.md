# MVP 完成要件ブレインストーム

**日付**: 2026-04-09
**ステータス**: approved

## 何を作るか

Report Design Studio v2 の MVP を完成させる。社内帳票作成ツールとして、テンプレートデザイン → データバインド → 印刷品質 PDF 出力のコアフローを完結させる。

### MVP スコープ定義

**対象ユーザー**: 社内の帳票作成・管理担当者
**コアフロー**: テンプレート作成 → API データ流し込み → 高品質 PDF 出力
**スコープ外**: 認証、マルチテナント、フォーム入力フロー、承認ワークフロー

## なぜこのアプローチか

**アプローチ A: PDF エンジン優先** を採用。

理由:
- PDF 出力品質がこのツールのコア価値に直結する
- API フロー（テンプレート JSON + データ JSON → PDF）は PDF エンジンに依存する
- 印刷品質 PDF をバックエンド（Java）で生成することで、API 経由の自動生成と UI プレビューの両方を実現できる
- 現行の html2canvas + jsPDF はピクセルベースであり、印刷品質には不十分

## キー決定事項

### 1. PDF 生成方式
- **決定**: Java バックエンドで PDF 生成（iText または Apache PDFBox）
- **理由**: テキスト検索可能なベクター PDF、高解像度印刷対応、API から直接呼び出し可能

### 2. データ流し込みフロー
- **決定**: API エンドポイント + UI プレビューの両方を MVP に含める
- **API フロー**: `POST /api/v2/pdf/generate` にテンプレート JSON + データ JSON → PDF バイナリ返却（Stateless）
- **UI フロー**: データソースパネルでデータをプレビュー表示し、PDF ダウンロード

### 3. テンプレート保存 / API 設計
- **決定**: MVP は localStorage 保存 + Stateless API
- **API 方式**: PDF 生成 API はテンプレート JSON をリクエストボディに含める（サーバー側にテンプレート保存不要）
- **理由**: サーバーサイドのテンプレート管理が不要になり、実装がシンプル。将来サーバー保存が必要になった場合は API を拡張すればよい

### 4. 認証
- **決定**: MVP では認証不要
- **理由**: 社内ネットワーク内での利用を想定

### 5. バグ対応
- **決定**: P2 バグ 17件を全件修正
- **理由**: 品質重視。既知の問題を残したままリリースしない方針

### 6. 日本語フォント戦略
- **決定**: PDF エンジンに日本語フォントを埋め込む
- **検討事項**: フォント選定（IPAex ゴシック/明朝などの OSS フォント or 商用フォント）、縦書き対応、ふりがな描画方式
- **理由**: 帳票の印刷品質に直結。フォント埋め込みなしではフォント置換が発生し表示が崩れる
